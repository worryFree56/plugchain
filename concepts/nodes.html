# Node type

# Full node

A full node is a program that validates blockchain transactions and blocks. Unlike light nodes that only process block headers and a small subset of transactions, running a full node requires more resources, but is required to become a validator. In addition, full nodes can also be used to provide stable low-latency blockchain APIs for applications.

# Validator full node

Plug Chain Hub based on Cosmos SDK (opens new window) and [Tendermint](https://tendermint .com/docs/introduction/what-is-tendermint.html) implementation, which relies on a set of validator nodes to maintain the security of the network. The role of the validator node is to run a full node and participate in consensus by broadcasting votes (containing ciphertext signed by its private key). Validator nodes submit new blocks in the blockchain and earn revenue to incentivize their work. They must also vote on proposals to participate in on-chain governance. The voting weight of validator nodes depends on their total plugcn stake.

# Candidate validator full node

Only the top 50 full nodes with the number of plugcn pledges can become validator nodes, and the rest will become candidate validator nodes. The ranking will be updated as the stake amount changes.

# Sentry node

Validator nodes need to ensure that the network can withstand a denial of service attack (DDoS).

A recommended way to mitigate these risks is to use sentinel nodes to build a network topology that protects validator nodes.

Validator nodes should only connect to full nodes they trust, such as those they maintain themselves or run by other validators they know in the community. Validator nodes will typically run in data centers. Most data centers offer direct connections to cloud providers. Validator nodes can connect to Sentry nodes in the cloud through such a network. This shifts the burden of DDoS directly from validator nodes to their sentinel nodes, and may require launching or activating new sentinel nodes when under attack to reduce the impact of the attack.

Sentry nodes can quickly start or change their IP addresses. Since the link from the validator node to the sentinel node is in the private IP space, hackers cannot directly attack the validator node through the network. This will ensure that block proposals and votes from validator nodes are always broadcast normally through sentinel nodes.

To configure the sentinel node architecture, you can follow the instructions below:

Validator nodes should edit their config.toml:

# Comma-separated list of nodes to keep persistent connections to
# If you don't want to expose private node information then don't add it to this list
persistent_peers = [list of sentinel peers]

# Disable P2P network peer-to-peer exchange
pex = false

Sentry nodes should edit their config.toml:

# Comma-separated list of node IDs to keep private (will not be broadcast to the P2P network)
# Example ID: 3e16af0cead27979e1fc3dac57d03df3c7a77acc

private_peer_ids = "node_ids_of_private_peers"

# seed node

A seed node provides a node with a list of peers to which the node can connect. When starting a node, you must provide at least one type of node to connect to the desired network. By providing a seed node, you will be able to populate your address quickly. Seed nodes will not remain peers, but will be disconnected from your node after a list of peers is provided.

WARNING

If you need to share the node for others to connect, you need to set external_address="node public IP:26656"

# Node configuration

# Validator node configuration:

Config Option Setting
pex false
persistent_peers list of sentry nodes
private_peer_ids none
unconditional_peer_ids optionally sentry node IDs
addr_book_strict false
double_sign_check_height 0

# Sentry node configuration:

Config Option Setting
pex true
persistent_peers validator node, optionally other sentry nodes
private_peer_ids validator node ID
unconditional_peer_ids validator node ID, optionally sentry node IDs
addr_book_strict false

# Seed node configuration:

Config Option Setting
addr_book_strict true
pex true
seed_mode true

# Server Configuration Recommendations

  1. API service nodes Synchronize blocks, broadcast transactions, and provide query functions for basic data on the chain
  • Minimum requirements
    • CPU: 2 cores
    • RAM: 4GB
    • Storage SSD: 1T
    • Bandwidth: 5Mbps
  • Recommended configuration:
    • CPU: 4 cores
    • RAM: 8GB
    • Storage SSD: 1T
    • Bandwidth: 5Mbps
  1. Validator nodes verify, produce blocks, and produce fast
  • Minimum configuration:
    • CPU: 4 cores
    • RAM: 8GB
    • Storage SSD: 1T
    • Bandwidth: 5Mbps
  • Recommended configuration:
    • CPU: 8 cores
    • RAM: 32GB
    • Storage SSD: 1T
    • Bandwidth: 20Mbps (the bigger the better)
  1. Seed node
    • CPU: 2 cores
    • RAM: 4GB
    • Storage SSD: 1T
    • Bandwidth: 4Mbps
